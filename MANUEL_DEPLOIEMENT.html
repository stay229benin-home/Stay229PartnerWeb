1. Comprendre ce qui a changé NOUVEAU

Avant de toucher quoi que ce soit, voici un récapitulatif clair des règles. Imprime cette page si tu veux : tu pourras t'y référer.

Pour les partenaires (influenceurs) — Refonte du 19 mai 2026

On est passés d'une commission fixe à une commission variable plafonnée, et on a ajouté un système de 5 niveaux pour motiver les partenaires les plus actifs.

NiveauFilleuls actifs *TauxPlafond / résaDurée payée **
11 à 53 %700 F4 mois
26 à 143 %700 F6 mois
315 à 293 %700 F9 mois
430 à 493,5 %850 F12 mois
Gold50 et +4 %1 000 Fà vie

* Filleul actif = filleul ayant fait au moins une réservation d'appartement confirmée. Un inscrit qui ne réserve jamais ne compte pas.
** La durée part de la 1ʳᵉ réservation confirmée parmi tous les filleuls du partenaire. Si le partenaire monte de niveau pendant la fenêtre, la durée du nouveau niveau s'applique automatiquement (donc la fenêtre s'allonge).

Comment ça se calcule concrètement : commission = min(montant_résa × taux, plafond).
Exemple niveau 1 : un filleul réserve à 15 000 F → commission = min(15000 × 3 %, 700) = 450 F.
Le même filleul réserve à 30 000 F → commission = min(30000 × 3 %, 700) = 700 F (le plafond).

Conditions pour retirer son argent

  • ≥ 3 filleuls liés au code du partenaire
  • ≥ 2 500 F CFA de commission cumulée
  • Paiements effectués manuellement chaque vendredi et samedi sur le numéro Mobile Money du partenaire
Mini-lexique Stay229 — Dans l'app, "court séjour" = vraie réservation d'appartement (paiement à la nuit ou aux 24H) ; "long séjour" = simple visite d'un logement à louer longue durée (2000 F forfaitaire). La commission partenaire ne tombe que sur les vraies réservations d'appart, pas sur les visites.

Codes promo personnalisés NOUVEAU

Le partenaire peut désormais générer plusieurs codes promo (un par plateforme, par campagne, etc.) depuis la page Codes promo de son espace. Pour chaque code, il peut renseigner facultativement la plateforme (Instagram / TikTok / WhatsApp / YouTube / Snapchat / X / LinkedIn / Bouche à oreille / Autre) et l'URL de son profil sur cette plateforme. Le code peut être auto-généré (8 caractères) ou choisi par le partenaire (par ex. JEAN229).

Pour les clients (filleuls)

Les clients qui s'inscrivent avec un code promo bénéficient automatiquement de 10 % de réduction sur leur toute première réservation d'appartement. C'est l'argument que les partenaires peuvent utiliser pour convaincre leur audience. La remise ne s'applique pas sur les frais de visite (2000 F) — par design, pour que les partenaires aient de quoi vendre du concret.

Le bug "invalid key" expliqué

Le fichier .env.local contenait la valeur fictive ta_clef_anon_ici pour la clé Supabase. Quand le site se lance, il essaie de se connecter à Supabase avec cette clé bidon → erreur "Invalid API key". Corrigé dans ce déploiement.

⚠️ Étape 0 — Désactiver "Confirm email" dans Supabase BUG FK

Symptôme : à l'étape "Créer mon compte partenaire", tu vois le message rouge :
insert or update on table "partners" violates foreign key constraint "partners_auth_user_id_fkey"

Cause : Supabase a deux protections "anti-énumération" qui empêchent le mini-site de connaître le vrai ID utilisateur tant que l'email n'est pas confirmé. Quand la confirmation par email est activée, le user n'apparaît dans auth.users qu'après clic sur le lien email — mais le mini-site essaie d'insérer dans partners immédiatement. La clé étrangère trouve aucune ligne correspondante → erreur.

Solution : on désactive "Confirm email" pour l'espace partenaires. C'est une simple bascule dans Supabase.

0.1 — Va sur le dashboard Supabase du projet.
0.2 — Dans la barre latérale gauche, clique sur l'icône Authentication (cadenas, en bas de la liste).
0.3 — Dans le sous-menu Authentication, clique sur Providers (ou Sign In / Up selon ta version de l'UI Supabase).
0.4 — Clique sur la ligne Email (généralement la 1ère ligne).
0.5 — Tu vois plusieurs interrupteurs. Décoche celui intitulé "Confirm email" (parfois noté "Require email confirmation"). L'interrupteur passe en gris.
0.6 — En bas du panneau, clique sur le bouton Save.
✓ C'est fait. Maintenant tout signup crée immédiatement le user dans auth.users et le mini-site peut créer la ligne partners sans erreur.

Si tu avais déjà essayé de t'inscrire avec un email…

L'email peut être déjà "fantôme" dans Supabase (créé mais non confirmé). Pour repartir propre avec le même email :

0.7 — Toujours dans Authentication, clique sur Users dans le sous-menu.
0.8 — Cherche l'email que tu as essayé (ex: doslinatoungbe7@gmail.com) dans la liste.
0.9 — Clique sur les 3 points (...) à droite de la ligne → Delete user. Confirme.
Astuce : tu peux aussi simplement utiliser un autre email pour ton premier test, et garder doslinatoungbe7@gmail.com pour un vrai compte plus tard.

8. En cas de problème

Erreur "Invalid API key" sur le site déployé

Tu as oublié de remplir les variables d'environnement sur Vercel. Va dans Vercel → ton projet → Settings → Environment Variables, ajoute NEXT_PUBLIC_SUPABASE_URL et NEXT_PUBLIC_SUPABASE_ANON_KEY avec les bonnes valeurs, puis dans l'onglet Deployments clique sur les 3 points ... du dernier déploiement → Redeploy.

Erreur "relation public.partners does not exist"

La migration 0014 n'a jamais été exécutée. Reviens en arrière et exécute-la (même procédure que 0017, dans le fichier 0014_referral_system.sql), puis enchaîne avec 0015 et 0017.

Erreur "cannot drop view partner_commissions because other objects depend on it"

Tu utilises une ancienne version du fichier 0017_referral_levels_and_multi_codes.sql. Récupère la dernière version (la section 6 du fichier doit commencer par drop view if exists public.partner_commission_totals; AVANT drop view if exists public.partner_commissions;) et relance.

Solution rapide : avant de coller le fichier, colle d'abord ces deux lignes dans le SQL Editor et clique Run : 
drop view if exists public.partner_commission_totals;
drop view if exists public.partner_commissions;
Puis colle le contenu du fichier 0017 et clique Run à nouveau.

Erreur "relation public.partner_promo_codes does not exist"

La migration 0017 n'a pas été exécutée. Refais l'étape 2 avec le fichier 0017_referral_levels_and_multi_codes.sql.

Les commissions ne s'affichent pas dans le dashboard

C'est normal au début : tant qu'un filleul n'a pas fait une réservation confirmée d'appartement (code CS…), aucune commission n'apparaît. Les visites (code LS…) ne génèrent jamais de commission.

Le bandeau "Niveau" affiche "Niveau 1" alors que mon partenaire a 30 filleuls

Le niveau est basé sur les filleuls actifs uniquement (au moins 1 résa confirmée d'appartement). Vérifie le champ confirmed_filleul_count dans la vue partner_commission_totals. Si la valeur est basse, c'est que la plupart des filleuls inscrits n'ont jamais réservé.

Le bouton "Codes promo" n'apparaît pas dans le menu

Le menu ne s'affiche que sur les pages connectées (Dashboard, Codes promo, Règles). Sur la page d'accueil /, /login et /signup, il est volontairement caché. Connecte-toi pour le voir apparaître.
